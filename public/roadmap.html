Sistema de Autenticação Intermediária

Gateway de cadastro, login e emissão de tokens JWT (acesso + atualização) para servir outras aplicações — com um serviço de domínio em Django/DRF (Postgres compartilhado) por trás, e todos os itens de prioridade Média entregues (RBAC, CSRF, cookies httpOnly, auditoria, verificação de e-mail, sair de todos os dispositivos). O que já foi feito

API · src/app/api/auth

  • POST /cadastro — cria usuário, senha hasheada, e-mail duplicado → 409, gera link de verificação (logado no console) e audita
  • POST /login — emite tokens ou desafio MFA (mfaObrigatorio), audita sucesso/falha
  • POST /atualizar — rotação de refresh token em transação, protegida por CSRF
  • POST /logout — revoga refresh token e limpa cookies, protegida por CSRF, audita
  • GET /me — usuário autenticado via cookie httpOnly ou Bearer (inclui papel, emailVerificado)
  • POST /verificar-email · GET /usuarios (admin) — verificação de e-mail e exemplo de RBAC

MFA & sessões · src/app/api/auth

  • POST /mfa/iniciar — gera segredo TOTP + QR code, protegida por CSRF
  • POST /mfa/confirmar / /desativar — ativa/desativa por código, protegidas por CSRF
  • POST /mfa/verificar — conclui o login com o código de 6 dígitos
  • GET /sessoes · DELETE /sessoes/[id] — listar/revogar sessões, protegida por CSRF
  • DELETE /sessoes — sair de todos os dispositivos, protegida por CSRF
  • POST /api/cron/limpar-tokens — limpeza protegida por CRON_SECRET

Lib · src/lib

  • token.ts — JWT (jose): acesso (RS256, claim papel), atualização, desafio MFA e verificação de e-mail
  • mfa.ts — segredo/QR TOTP (otpauth + qrcode) e verificação
  • sessao.ts — emissão de tokenAcesso/tokenAtualizacao/csrfToken em cookies httpOnly
  • cookies.ts — cookies httpOnly (acesso, atualização) + CSRF não-httpOnly
  • csrf.ts — geração/validação do token CSRF (double-submit cookie)
  • auditoria.ts — registro de eventos com IP/user-agent, best-effort
  • senha.ts — hash/verificação com bcryptjs
  • validacao.ts — schemas Zod
  • clienteAuth.ts — sem sessionStorage, cookies httpOnly + header CSRF automático
  • autenticar.ts — aceita token via cookie httpOnly ou Bearer

Dados

  • Prisma + Postgres local — Usuario, TokenAtualizacao, LogAuditoria
  • Usuario.mfaAtivado / mfaSecret / emailVerificado / papel (enum Papel)
  • Migrado de SQLite sem perda de dados existentes

Frontend

  • Páginas: home, /login (2ª etapa MFA), /cadastro, /dashboard, /verificar-email
  • Dashboard com seção Segurança: sessões ativas (individual + todas de uma vez) + ativar/desativar MFA + indicador de e-mail verificado
  • /dashboard/projetos e /dashboard/projetos/[id] — CRUD de projetos e tarefas (Django)
  • proxy.ts redireciona rotas protegidas / somente-visitante
  • Sem sessionStorage — nenhum token guardado manualmente pelo cliente

Serviço de domínio · django/

  • Token de acesso migrado de HS256 para RS256 — chave privada só no Next.js, pública compartilhada
  • comum/autenticacao.py — classe DRF que valida o JWT (algorithms=["RS256"] fixo, via header Bearer ou cookie tokenAcesso) e usa os claims sub/papel como identidade, sem model de usuário próprio
  • App tarefas com entidades reais: Projeto e Tarefa (status, prazo, isolados por usuário)
  • next.config.ts encaminha /api/dominio/* para o Django via rewrite — mesma origem, sem CORS, cookies repassados transparentemente
  • Postgres compartilhado com o Next.js (mesma instância local), database própria autenticacao_dominio
  • ProtegidoContraCsrf — mesma proteção CSRF double-submit-cookie do lado Next.js, nas mutações de ProjetoViewSet/TarefaViewSet
  • 23 testes automatizados (pytest-django) — JWT válido/expirado/adulterado/confusão de algoritmo/cookie/papel + CRUD/isolamento/CSRF

Segurança já presente

  • Senha hasheada (bcrypt, 12 rounds)
  • Refresh token hasheado no banco — nunca gravado em texto puro
  • Rotação de refresh token a cada renovação, com erros inesperados tratados/logados
  • Access token e refresh token em cookies httpOnly — não acessíveis a JavaScript, mitiga roubo via XSS
  • Access token de vida curta (15 min), assinado com RS256
  • Proteção CSRF explícita (double-submit cookie) em toda mutação autenticada por cookie, Next.js e Django
  • Verificação em duas etapas (TOTP) opcional por usuário
  • Verificação de e-mail (token stateless, link logado no console em dev)
  • RBAC mínimo (Usuario.papel, claim no token, rota de exemplo restrita a admin)
  • Gestão de sessões ativas (individual ou todas de uma vez)
  • Logs de auditoria (login sucesso/falha, cadastro, logout — IP e user-agent)
  • Job de limpeza de tokens expirados/revogados antigos
  • Segredos JWT gerados aleatoriamente e documentados (não mais placeholders)
  • Serviço Django valida algoritmo fixo (RS256) — bloqueia ataque de confusão de algoritmo

proxy.tsMantido como está

Checagem é otimista (só valida assinatura, não revogação no banco).

Decisão consciente: a doc oficial do Next.js 16 recomenda não usar o Proxy como solução completa de autorização/sessão — a validação real já acontece nas rotas de API, onde os dados de fato são lidos/gravados. Adicionar consulta ao banco a cada carregamento de página iria contra essa recomendação.

Por onde começar

Restam só 4 itens, todos de prioridade Alta: recuperação de senha, rate limiting e testes automatizados do Next.js (o lado Django já tem) são os de maior impacto antes de qualquer uso além de ambiente de desenvolvimento.

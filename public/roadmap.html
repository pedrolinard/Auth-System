Concluído

O que já foi feito

API · src/app/api/auth

  • POST /cadastro — cria usuário, senha hasheada, e-mail duplicado → 409
  • POST /login — emite access token (15 min) + refresh token (30 dias)
  • POST /atualizar — rotação de refresh token em transação
  • POST /logout — revoga refresh token e limpa cookie
  • GET /me — usuário autenticado via Bearer token

Lib · src/lib

  • token.ts — JWT (jose) + hash SHA-256 do refresh token
  • senha.ts — hash/verificação com bcryptjs
  • cookies.ts — cookie httpOnly / secure / sameSite=lax
  • validacao.ts — schemas Zod
  • clienteAuth.ts — token em sessionStorage, refresh automático em 401
  • autenticar.ts — extração/validação de Bearer token

Dados

  • Prisma + SQLite — modelos Usuario e TokenAtualizacao
  • Migração inicial aplicada

Frontend

  • Páginas: home, /login, /cadastro, /dashboard
  • proxy.ts redireciona rotas protegidas / somente-visitante

Segurança já presente

  • Senha hasheada (bcrypt, 12 rounds)
  • Refresh token hasheado no banco — nunca gravado em texto puro
  • Rotação de refresh token a cada renovação
  • Cookie httpOnly para o refresh token
  • Access token de vida curta (15 min)

Melhorias no que já existe

atualizar/route.ts

Garantir tratamento/log de erros inesperados do Prisma na transação de rotação de token.

clienteAuth.ts → obterUsuarioAtual

A recursão após o refresh pode entrar em loop se o refresh funcionar mas /me continuar falhando por outro motivo (ex. 500).

Limitar a 1 nova tentativa.

dashboard/page.tsx

Expõe o JWT de acesso na tela — ok para debug, remover antes de produção real.

login/route.ts

Mensagens de erro já evitam vazar se é e-mail ou senha errada — bom padrão, manter em outros pontos.

proxy.ts

Checagem é otimista (só valida assinatura, não revogação no banco).

Considerar validar contra o banco em rotas realmente sensíveis.

Por onde começar

Recuperação de senha, rate limiting e testes automatizados são os três itens de maior impacto antes de qualquer uso além de ambiente de desenvolvimento.
